Setting up authentication & CORS

How to connect a narrow-casting screen to the i-Reserve REST API.

1. How the connection works

Each example fetches data with a GET request to the REST API:

GET https://YOURENVIRONMENT.i-reserve.net/api/rest/booking/filter?columns=...&search=...
Authorization: Basic <base64(user:password)>
  • booking/filter → reservations/bookings (day schedule, room signs, slideshow).
  • event/filter → public events with availability (events grid).

The columns parameter determines which fields are returned; the search parameter is the filter (see §5).

2. Create an API user (read-only!)

⚠️ Important: with narrow casting the login details are readable in the HTML (the browser sends them along). So never use an administrator account, but a dedicated user with read-only rights on exactly the products the screen needs.
  1. Log in to the i-Reserve admin panel as an administrator.
  2. Go to Users / Staff and create a new user, e.g. API_signage.
  3. Give this user a role/permission profile with read-only rights on bookings/events. No write, delete or admin rights.
  4. Where possible, restrict access to only the relevant products/rooms.
  5. Choose a strong, unique password and note it. You will enter it in config.js.
💡 Optionally create a separate user per location/screen group, so you can revoke credentials per screen without affecting the rest.

3. Authentication in the HTML (Basic Auth)

The SDK handles this automatically. You only fill in your details in config.js:

auth: { username: "API_signage", password: "YOUR_PASSWORD" },

The helper (shared/ireserve.js) builds the header from this:

"Authorization": "Basic " + btoa(username + ":" + password)

If an endpoint needs no authentication (some public event endpoints), set auth: null.

4. Allow CORS in the admin panel ⭐

This is the step most often forgotten. The browser blocks requests to a different domain than the page came from by default. A narrow-casting page usually runs as a file (file://) or on a local box — a different origin than https://YOURENVIRONMENT.i-reserve.net. So the API must explicitly allow this origin to make requests (CORS = Cross-Origin Resource Sharing).

Which origin should you allow?

How does the screen run?Origin to whitelist
As a loose file (file://.../index.html)null
From a local web serverhttp://localhost or http://<ip-address>
Hosted on a (sub)domainhttps://signage.yourdomain.com
With file:// the browser literally sends the origin null. So that must be in the whitelist. If that's not possible, host the page via a mini web server (see §7).

Where in the admin panel

  1. Open the admin panel → Settings → API / Integrations (or Security / CORS — the exact name may differ per version).
  2. Find the setting "Allowed origins" / "CORS allowed origins".
  3. Add the origin(s) from the table above, one per line, for example: 
    null
https://signage.yourdomain.com
  4. Save. The API then returns the correct Access-Control-Allow-Origin header.
Avoid * (allow everything) in production. Whitelist only the origins you actually use. * also does not work together with Basic Auth credentials in some browsers.

5. The filter (search DSL)

The search parameter uses relative dates and simple logic:

ExampleMeaning
startdatum >= "0d" AND startdatum <= "0d"today only
startdatum >= "0d" AND startdatum <= "7d"today through 7 days ahead
startdatum >= "-1d"from yesterday
... AND product IN (101,102,103)limit to these product IDs

"0d" = today, "7d" = in 7 days, "-1d" = yesterday. Combine with AND / OR. You find the product IDs in the admin panel on the relevant product/room.

The columns parameter determines the fields in the response, for example OBJ_TXT_CUSTOM01,RES_CUSTOM02,RES_FROMTIME,RES_TILTIME.

6. Testing that it works

First test outside the browser with curl (no CORS restriction, so purely the auth/filter check):

curl -u "API_signage:YOUR_PASSWORD" \
  'https://YOURENVIRONMENT.i-reserve.net/api/rest/booking/filter?columns=RES_CUSTOM02,RES_FROMTIME,RES_TILTIME&search=startdatum%20%3E%3D%20%220d%22'
  • JSON returned → auth and filter are fine. Then open the screen in the browser.
  • 401 Unauthorized → username/password or rights are wrong (§2).
  • Empty in the browser but data with curl → almost certainly CORS (§4). Open the DevTools console (F12); a CORS message confirms it.

The bottom-left of every screen shows a status line with the latest fetch status/error.

7. Alternative: host via a mini web server

If you can't whitelist origin null, run the page via a local web server. Then you have a clean http://localhost origin:

# in the example's folder
python3 -m http.server 8080
# then open http://localhost:8080 on the screen

Then whitelist http://localhost:8080 (or the IP) in §4. The SDK also ships demo/server.js (node demo/server.js) which serves the repo and a mock API.

8. Security checklist

  • Dedicated API user, read-only, limited to the right products.
  • Strong, unique password; rotate periodically.
  • Only the needed origins in the CORS whitelist (no *).
  • No admin account in config.js.
  • When in doubt: put your own reverse proxy in between that adds the credentials server-side, so they are not in the HTML.
